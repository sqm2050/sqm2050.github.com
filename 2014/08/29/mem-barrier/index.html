变量当作开关，被编译器优化

    while (pgsreg->ctrl_sw.flag_decode_lte) {
    }
  • flag_decode_lte 没有加上volatile关键字修饰，如果flag_decode_lte被设置，就会进入死循环。如果在被设置为零也会跳出循环。
    10001928:    e896034a     bbit1    a0,0x16,10002654 <zxmd_main_process+0xeec>
    ...
    10002654:    08000995     j    10002654 <zxmd_main_process+0xeec>
  • 加上volatile关键字之后
  •   10001928:    dc620020     ld    v0,32(v1)
  1000192c:    e856fffe     bbit1    v0,0x16,10001928 <zxmd_main_process+0x1c0>

  • 加上内存栅栏

      while (pgsreg->ctrl_sw.flag_decode_lte) {
      asm volatile ("": : : "memory");
  }

汇编码：

    10001930:    dc825d88     ld    v0,23944(a0)
    10001934:    dc430020     ld    v1,32(v0)
    10001938:    e876fffd     bbit1    v1,0x16,10001930 <zxmd_main_process+0x1c8>

效果比volatile还要明显，ctrl_sw变量的地址也是重新获取的。

  • 这里在介绍一下sync同步

  • 如上在内存栅栏中没有添加sync指令，只是破坏性指示中有“memory”，这样仅能保证没去做load指令，但是在有cache的系统中不能保证cache的同步，所以可以加上sync指令，加强同步。

      while (pgsreg->ctrl_sw.flag_decode_lte) {
      asm volatile ("sync": : : "memory");
  }

汇编码

    10001930:    0000000f     sync
    10001934:    dc825d88     ld    v0,23944(a0)
    10001938:    dc430020     ld    v1,32(v0)
    1000193c:    e876fffc     bbit1    v1,0x16,10001930 <zxmd_main_process+0x1c8>

  • 可是如果只是加上sync指令，它指示cache需要同步，但是不能防止gcc对代码的优化
    如：

      while (pgsreg->ctrl_sw.flag_decode_lte) {
      asm volatile ("sync": :);
  }

如果没有破坏性指示最后一个：不需要，否则编译不通过。
